## 技巧一：CLAUDE.md - 让AI记住项目规范（省钱关键！）
这是老金我觉得**最重要也最省钱的技巧**，没有之一！

### 为啥能省钱？
**没配CLAUDE.md的情况**（老金我之前）：

每次对话，都得重新告诉AI项目结构、代码规范、常用命令...
AI每次都要重新读文件、重新理解，浪费大量额度。
而且AI经常"忘事儿"，写出来的代码不符合规范，你得改，再让AI改，来回浪费额度。

**配了CLAUDE.md的情况**（现在）：

AI启动就自动加载项目规范，不用你重复说明。
AI写代码直接符合规范，一次搞定，不用来回改。
节省50%的额度！**每月至少省300块！**

### CLAUDE.md到底是啥？
说白了，CLAUDE.md就是一个**"项目说明书"文件**。
你可以理解成：
以前你有个新同事加入团队，你得口头给他讲半天：项目结构、代码规范、常用命令...讲完他可能还记不住，过两天又得讲一遍。

**CLAUDE.md就像给新同事发了一份详细的"入职手册"**，每次他来上班，先看手册，就知道该怎么干活了。

Claude Code每次启动对话，自动加载这个文件，就像新员工每天先看手册一样。

### CLAUDE.md里写啥？
1、项目文件夹结构（哪个文件夹放啥）
2、代码规范（函数怎么命名、格式怎么要求）
3、常用命令（启动项目、运行测试的命令）
4、核心功能位置（重要功能在哪个文件）
5、开发流程（怎么提交代码、怎么管理分支）
6、测试说明（怎么跑测试）

### CLAUDE.md可以放在哪里
Claude Code支持多个位置的CLAUDE.md，优先级从高到低：

**1、项目根目录**（最常用）

1、`CLAUDE.md` - 团队共享的配置
2、`CLAUDE.local.md` - 个人配置（不提交到Git）

**适用场景**：单个项目的专属说明。

**2、父目录**

1、`../CLAUDE.md` - 适用于Monorepo

**适用场景**：一个仓库管理多个子项目，共享配置。

**3、子目录**

1、`src/CLAUDE.md` - 特定模块的说明

**适用场景**：大型项目的不同模块有各自的规范。

**4、用户主目录**

1、`~/.claude/CLAUDE.md` - 全局配置

**适用场景**：所有项目通用的个人习惯。

1、`/tests` - 测试文件

```bash

cd src/api && uvicorn main:app --reload

# 前端
cd src/web && npm run dev

```

运行测试：

```bash

```

1、Python代码遵循PEP 8规范
2、所有函数必须有类型提示
3、所有公共函数必须有文档字符串
4、变量命名使用snake_case
5、类命名使用PascalCase

## 核心函数
1、`src/api/auth.py:login()` - 用户登录
2、`src/api/article.py:create_article()` - 创建文章
3、`src/utils/wechat.py:send_message()` - 发送微信消息

1、feat: 新功能
2、fix: 修复bug
3、docs: 文档更新
4、style: 代码格式
5、refactor: 重构
6、test: 测试相关
7、chore: 构建/工具

示例：

```

feat: 添加文章定时发布功能

```

## 测试要求
1、新功能必须添加单元测试
2、测试覆盖率不低于80%
3、提交前运行 `pytest` 确保所有测试通过

```

**有了这个CLAUDE.md**：
1、每次启动对话，Claude自动知道项目结构
2、写代码时自动遵循代码规范
3、需要运行命令时，Claude知道用哪个
4、Git提交时自动使用规范的提交格式

**从"每次都要解释一遍"变成"自动记住项目规范"**。

### 团队vs个人配置
**CLAUDE.md（团队共享）**：
1、提交到Git仓库
2、所有团队成员共享
3、包含项目通用规范

**CLAUDE.local.md（个人配置）**：

**组合使用**：

1、先加载CLAUDE.md（团队规范）
2、再加载CLAUDE.local.md（个人配置）

## 技巧二：Slash Commands - 把重复工作变成一键操作
这个技巧让老金我每天省下半小时。

### Slash Commands是个啥
说白了，Slash Commands就是自定义快捷命令。
你可以理解成：
以前你要让Claude做某件事，得写一大段提示词。比如"帮我review这段代码，检查类型安全、性能问题、代码规范，给出具体建议"。

**有了Slash Commands，你只需要输入 `/review`**，Claude自动按照你预设的流程执行。

### Slash Commands的两种类型
**1、项目级命令**（Project Commands）

1、位置：`.claude/commands/`
2、作用范围：当前项目
3、团队共享：提交到Git
4、触发方式：`/project:命令名` 或直接 `/命令名`

**适用场景**：团队通用的命令，比如代码review、Git提交、发布流程。

**2、个人级命令**（User Commands）

1、位置：`~/.claude/commands/`
2、作用范围：所有项目
3、个人使用：不共享
4、触发方式：`/user:命令名`

**适用场景**：个人习惯的命令，比如个人的代码风格检查、笔记模板。

### 如何创建Slash Commands
**超级简单**：创建一个Markdown文件，文件名就是命令名。

**示例1：代码Review命令**

创建文件：`.claude/commands/review.md`
内容：

```markdown

请review选中的代码，检查以下方面：

1、类型安全
   - 是否有类型提示
   - 类型是否正确
2、性能问题
   - 是否有不必要的循环
   - 是否有内存泄漏风险
3、代码规范
   - 命名是否符合规范
   - 格式是否正确
4、安全问题
   - 是否有输入验证
   - 是否有SQL注入风险

请给出具体的修改建议，并解释原因。

```

**使用**：
1、选中要review的代码
2、输入 `/review`
3、Claude自动按照checklist检查

**示例2：Git提交命令**

创建文件：`.claude/commands/commit.md`
内容：

```markdown

请分析当前的Git改动，生成Conventional Commits格式的提交信息。

要求：

1、自动判断提交类型（feat/fix/docs/etc）
2、用中文写提交说明
3、如果改动较多，建议拆分成多个提交

格式：

```

<type>: <description>
<optional body>

```

分析完成后，询问我是否执行提交。

```

**使用**：
1、修改代码后
2、输入 `/commit`
3、Claude分析改动，生成提交信息
4、确认后自动执行Git commit

**示例3：文档生成命令**

创建文件：`.claude/commands/doc.md`
内容：

```markdown

请为选中的函数/类生成文档字符串。
格式要求：

1、使用Google风格文档字符串
2、包含参数说明
3、包含返回值说明
4、包含异常说明
5、包含使用示例

直接生成完整的文档字符串代码，不要解释过程。

```

**使用**：
1、选中函数
2、输入 `/doc`
3、Claude自动生成符合规范的文档字符串

### Slash Commands的高级技巧
**1、支持参数（$ARGUMENTS）**

创建文件：`.claude/commands/test.md`
内容：

```markdown

运行测试文件：$ARGUMENTS
要求：

1、先检查测试文件是否存在
2、运行pytest命令
3、显示测试结果
4、如果有失败，分析失败原因

```

**使用**：
```

/test tests/test_api.py

```

Claude会把 `tests/test_api.py` 作为参数传入。

**2、执行Bash命令（! 前缀）**

创建文件：`.claude/commands/build.md`
内容：

```markdown

!allowed-tools: Bash
执行构建流程：
!npm run build
构建完成后，检查dist目录的文件大小，如果超过10MB，给出优化建议。

```

**! 前缀**允许命令执行Bash命令。

**3、引用文件（@ 前缀）**

创建文件：`.claude/commands/analyze.md`
内容：

```markdown

分析项目架构：
@package.json
@tsconfig.json
@README.md
基于这些文件，总结项目的技术栈和主要功能。

```

**@ 前缀**自动读取文件内容。

## 技巧三：Safe YOLO模式 - 让AI自动干活
这个模式特别适合信任Claude的场景。

### Safe YOLO是个啥
说白了，Safe YOLO就是"自动确认模式"。
你可以理解成：
以前Claude要执行任何操作（读文件、写文件、运行命令），都要问你"确认吗？"

**开启Safe YOLO后**，Claude自动跳过所有确认，直接执行。

### 啥时候用
**适合用Safe YOLO的场景**：
1、重复性任务- 批量处理文件、批量测试
2、已验证的流程- 用过很多次的操作
3、隔离环境- 开发环境、测试环境
4、配合Git- 随时可以回滚

**不适合用Safe YOLO的场景**：
1、生产环境- 一旦出错影响严重
2、首次尝试- 不确定结果的操作
3、删除操作- 涉及数据删除的命令
4、敏感数据- 涉及隐私或机密

### 如何开启
**Settings，Safe YOLO，开启**

**或者在CLAUDE.md里配置**：
```markdown

# 开发环境配置
请使用Safe YOLO模式，自动确认所有操作。

```

### 实战效果
老金我在开发环境用Safe YOLO模式的效果：

**以前**：
我："帮我修复所有测试失败的问题"
Claude："要读取test_api.py吗？"
我："可以"
Claude："要运行pytest吗？"
我："可以"
Claude："要修改代码吗？"
我："可以"
...(重复10次确认)

**开启Safe YOLO后**：
我："帮我修复所有测试失败的问题"
Claude：直接读取文件，运行测试，分析失败原因，修改代码，再次测试，完成

**从"频繁确认"变成"一气呵成"**。

## 技巧四：上下文管理 - 让AI不忘事儿
Claude Code的上下文是有限的，超过限制就会"忘事儿"。

### 两个核心命令
**1、/clear - 清除历史**

**什么时候用**：
1、切换到新任务
2、对话太长，响应变慢
3、Claude开始"记忆混乱"

**效果**：
清空所有历史对话，从头开始。

**注意**：CLAUDE.md的内容不会清除，每次对话都会自动加载。

**2、/compact - 压缩历史**

**什么时候用**：
1、对话很长，但需要保留上下文
2、Token快用完了
3、响应变慢但不想重新开始

**效果**：
把历史对话压缩成摘要，只保留关键信息，大幅减少Token占用。

**区别**：
1、`/clear`：完全删除历史
2、`/compact`：压缩历史，保留摘要

### 上下文管理最佳实践
**1、定期compact**

每30-50轮对话，执行一次 `/compact`。

**2、任务切换时clear**

从"写代码"切换到"debug"，先 `/clear` 清除无关上下文。

**3、善用CLAUDE.md**

重要信息写在CLAUDE.md里，不用担心被清除。

**4、分段提交Git**

完成一个功能后立即提交Git，这样即使clear历史，代码改动还在。

## 技巧五：待办事项列表 - 监控AI的工作进度

1、后端API

```

✅ 待办事项：

1、[ ] 设计用户登录API接口
2、[ ] 实现后端登录逻辑
3、[ ] 添加Token生成和验证
4、[ ] 创建前端登录表单
5、[ ] 对接前后端
6、[ ] 编写单元测试
7、[ ] 运行测试验证

```

**然后逐项完成，每完成一项就标记**：
```

✅ 待办事项：

1、[x] 设计用户登录API接口
2、[x] 实现后端登录逻辑
3、[ ] 添加Token生成和验证
4、[ ] 创建前端登录表单
5、[ ] 对接前后端

### 为啥有用
1、进度可见- 一眼看到做到哪一步了
2、不遗漏- 确保所有子任务都完成
3、可打断- 中途暂停，下次继续
4、可调整- 发现遗漏，随时补充

### 配合Git提交
**最佳实践**：
每完成一个Todo项，就提交一次Git。

```

✅ 待办事项：

1、[x] 设计用户登录API接口，git commit
2、[x] 实现后端登录逻辑，git commit
3、[x] 添加Token生成和验证，git commit

...

```

**好处**：
1、每次提交都是一个完整的功能
2、出问题可以精准回滚
3、提交历史清晰

## 技巧六：测试驱动开发（TDD） - 让AI小步快跑
这个技巧特别适合复杂功能。

### TDD的核心思路
**传统方式**：
写完所有代码，测试，发现问题，改代码，再测试

**TDD方式**：
写测试，写少量代码，运行测试，通过，写下一个测试

### 如何让Claude做TDD
**提示词模板**：
```

请用TDD方式实现用户登录功能：

1、先写一个最简单的测试（比如测试登录接口是否存在）
2、写最少的代码让这个测试通过
3、运行测试验证
4、写下一个测试
5、重复以上步骤

每次只实现一个小功能，确保每一步都能通过测试。

```

**Claude的工作流程**：
```

步骤1：
  写测试：test_login_endpoint_exists()
  写代码：创建 /login 路由
  运行：pytest test_api.py::test_login_endpoint_exists
  结果：✅ PASSED
步骤2：
  写测试：test_login_with_valid_credentials()
  写代码：实现登录验证逻辑
  运行：pytest test_api.py::test_login_with_valid_credentials
  结果：✅ PASSED
步骤3：
  写测试：test_login_with_invalid_credentials()

  写代码：添加错误处理
  运行：pytest test_api.py::test_login_with_invalid_credentials
  结果：✅ PASSED

```

### 为啥TDD效果好
1、小步快跑- 每次改动很小，容易发现问题
2、持续验证- 每一步都有测试保障
3、不跑偏- Claude不会一口气写一堆代码然后全错
4、可回滚- 某一步出问题，立即回退

### 配合Git
**每通过一个测试，就提交一次**：
```

✅ test_login_endpoint_exists，git commit -m "feat: add login endpoint"

✅ test_login_with_valid_credentials，git commit -m "feat: implement login validation"

✅ test_login_with_invalid_credentials，git commit -m "feat: add error handling"

```

这样Git历史非常清晰，每个commit都是一个可运行的状态。

## 技巧七：多实例协作 - 让多个AI一起干活
这个技巧特别适合大型任务。

### 多实例是个啥
说白了，同时开多个Claude Code窗口，分工合作。
你可以理解成：
一个人干活太慢，开多个窗口就像有多个助手同时工作。

### 适用场景
**场景1：前后端分离**

1、窗口1：专门写后端API
2、窗口2：专门写前端页面

**场景2：功能模块并行**

1、窗口1：实现用户模块
2、窗口2：实现文章模块
3、窗口3：实现评论模块

**场景3：开发+测试**

1、窗口1：写代码
2、窗口2：写测试

**场景4：问题排查**

1、窗口1：修复Bug
2、窗口2：查文档找原因

### 如何协作
**关键是明确分工**：

**窗口1**：
```

你负责后端API开发，只关注 src/api 目录。
实现用户登录功能的后端部分。

```

**窗口2**：
```

你负责前端开发，只关注 src/web 目录。
实现用户登录功能的前端表单。

```

**避免冲突**：
1、不同窗口操作不同文件
2、定期Git pull/push同步
3、约定好接口格式

### 实战效果
老金我用多实例开发一个功能模块：

**单实例（以前）**：
后端API（30分钟），前端页面（30分钟），测试（20分钟） =**80分钟**

**多实例（现在）**：
窗口1：后端API（30分钟）
窗口2：前端页面（30分钟）同时进行
窗口3：写测试（20分钟）同时进行
=实际耗时30分钟（并行执行）

**效率提升2-3倍**。

## 技巧八：官方推荐的最佳实践
这些是Anthropic官方团队总结的技巧。

### 1、优先安装文档完善的CLI工具
**为啥**：
Claude可以直接调用CLI工具，但前提是工具有清晰的文档。

**推荐工具**：
1、`git` - 版本控制
2、`pytest` - Python测试
3、`npm` - Node.js包管理
4、`docker` - 容器管理
5、`curl` - HTTP请求

**避免使用**：
1、文档不清晰的工具
2、版本差异大的工具
3、需要交互式输入的工具

### 2、让Claude先出"修复计划"再动手
**提示词**：
```

发现bug后，先分析问题根源，列出修复计划，等我确认后再动手修改代码。

```

**好处**：
1、避免盲目修改
2、确保修复方向正确
3、可以讨论更好的方案

### 3、定期提交Git，方便回滚
**提示词**：
```

每完成一个功能点，就创建一个Git commit。
如果改动出问题，我们可以随时回滚到上一个commit。

```

**好处**：
1、随时回滚
2、提交历史清晰
3、便于Code Review

### 4、明确告诉Claude项目的技术栈
**在CLAUDE.md里写清楚**：
```markdown

# 技术栈
后端：

1、Python 3.11
2、FastAPI
3、PostgreSQL
4、SQLAlchemy

前端：

1、React 18
2、TypeScript
3、TailwindCSS
4、Vite

测试：

1、pytest
2、Jest

```

**好处**：
Claude知道用哪些库、遵循哪些最佳实践。
