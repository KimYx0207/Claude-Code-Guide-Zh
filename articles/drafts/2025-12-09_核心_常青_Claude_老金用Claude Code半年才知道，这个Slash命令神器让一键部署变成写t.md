## Slash Commands到底是什么？
说白了，Slash Commands就是**你给AI写的快捷键说明书**。
你可以理解成：
以前你家里买了个扫地机器人，每次都得手动按键：先扫客厅、再扫卧室、最后扫厨房，按3次。

现在你在遥控器上设置一个"全屋打扫"按钮，按一次，机器人自动按顺序干完这3件事。

**Slash Commands就是这个"全屋打扫"按钮**。

具体怎么做？

1、在项目文件夹里创建一个叫`.claude`的文件夹
2、在`.claude`里再创建一个`commands`文件夹
3、在`commands`里创建一个txt文件，改后缀名为`.md`，文件名就是命令名
4、在文件里写上你要AI执行的步骤
5、在Claude Code里敲 `/文件名`，AI就自动按你写的步骤干活

比如你创建了`.claude/commands/deploy.md`，内容是"部署项目到生产环境的步骤"，那你在Claude Code里敲`/deploy`，AI就自动照着你写的步骤执行。

**没配Slash Commands的情况（老金我之前每天的痛苦）**：
每次都得手动输一大段：

```

你: "帮我部署到生产环境，记得先运行测试、构建、然后推送到服务器，
     还要重启服务，对了，别忘了备份数据库..."
AI: "好的，我一步步来..."
（然后你心里七上八下：它会不会漏步骤？会不会搞错顺序？）

## 创建你的第一个Slash Command（3分钟）
老金我手把手教你，包你学会！

### 步骤1：创建命令文件夹
**Windows用户**：
1、打开你的项目文件夹（就是写代码的那个文件夹）
2、右键，新建文件夹，命名为`.claude`（前面有个点，别漏了）
3、进入`.claude`文件夹，再新建一个文件夹，命名为`commands`

**Mac/Linux用户**：
在项目文件夹里打开命令行，输入：

```bash

mkdir -p .claude/commands

```

### 步骤2：创建命令文件
**Windows用户**：
1、进入`.claude/commands`文件夹
2、右键，新建文本文档
3、命名为`deploy.txt`
4、重命名，把`.txt`改成`.md`（系统可能会提示你"确定要更改吗"，点确定）

**Mac/Linux用户**：
```bash

touch .claude/commands/deploy.md

```

### 步骤3：写命令内容
用任何文本编辑器（记事本、VS Code、Notepad++都行）打开`deploy.md`，写入下面这些内容：

```markdown

---

name: deploy
description: 部署项目到生产环境

---

请按以下步骤部署项目到生产环境：

1、运行测试确保代码质量
2、生产构建
3、备份当前生产数据库
4、推送代码到生产服务器
5、安装生产依赖
6、重启服务
7、验证部署成功

**重要**：每步执行前向我确认，如果某步失败立即停止后续步骤。

```

**解释一下**：
1、前面三行横线之间的是"命令信息"，告诉AI这个命令叫啥、干啥用
2、下面就是你要AI执行的步骤，**用大白话写清楚就行**
3、不用写具体代码命令，AI会自己判断该执行什么命令

**老金我的经验**：命令文件里写得越详细，AI执行得越准确。但也不用太专业，就像跟朋友说话一样，说清楚要做什么就行。

### 步骤4：使用命令
保存文件，回到Claude Code，直接敲：

```

/deploy

```

AI立即按照你写的步骤开始干活！

**就这么简单！从此告别重复输入！**

## 老金的5个必备Slash Commands

### 命令1：/code-review - 代码审查
**文件**：`.claude/commands/code-review.md`

```markdown

---

name: code-review
description: 全面审查当前分支的代码变更

---

请对当前分支进行全面代码审查：

1、查看改动文件

```bash

git diff --name-only main...HEAD

```

2、查看具体改动

```bash

git diff main...HEAD

```

3、审查要点：
1、代码质量：是否遵循CLAUDE.md的编码规范？
2、潜在bug：是否有明显的逻辑错误或边界条件问题？
3、性能问题：是否有不必要的循环、重复计算？
4、安全隐患：是否暴露敏感信息、SQL注入风险？
5、测试覆盖：重要逻辑是否有单元测试？
6、可维护性：代码是否容易理解和修改？
4、输出格式：

```markdown

## 代码审查报告

### 做得好的地方
1、[列出优点]

### 需要改进的地方
1、[问题1]（位置：文件名:行号）
2、[问题2]（位置：文件名:行号）

### 发现的潜在bug
1、[bug描述]（位置：文件名:行号）

### 优化建议
1、[建议]

```

5、如果发现严重问题（安全漏洞、数据丢失风险），立即高亮提醒我！

```

**使用场景**：提交PR前快速自检

3、如果改了API相关文件，运行集成测试

1、显示失败的测试用例

**使用场景**：改代码后快速验证

### 命令5：/doc - 自动生成文档

```markdown

name: doc
description: 为新增代码自动生成文档

---

请为新增的代码生成文档：

1、找出新增的函数、组件、类

```bash

git diff main...HEAD

```

2、为每个新增的代码块生成文档：

**函数文档格式**：
```typescript

/**

 * [函数功能简述]
 *
 * @param {类型} 参数名 - 参数说明
 * @returns {类型} 返回值说明
 * @throws {ErrorType} 错误类型 - 抛出条件
 *
 * @example
 * ```typescript
 * [使用示例]
 * ```
 */

```

**组件文档格式**：
```typescript

/**

 * [组件功能简述]
 *
 * @component
 * @param {Object} props - 组件属性
 * @param {类型} props.xxx - 属性说明
 *
 * @example
 * ```tsx
 * <ComponentName prop1="value" />
 * ```

1、新增的API需要在README记录
2、新增的重要功能需要说明
3、破坏性变更需要Migration Guide

```markdown

```

## 常见问题

### Q1：命令文件必须是markdown格式吗？
**是的**，必须是`.md`文件。

但你可以在markdown里写任何内容：

1、bash命令
2、指令描述
3、代码示例
4、检查清单

### Q2：命令可以调用其他命令吗？
**可以**，在命令里写`/other-command`就行。

比如：

```markdown

---

name: full-check
description: 完整检查

---

请依次执行：

1、/test
2、/fix-lint
3、/code-review

全部通过后提示我可以提交了。

```

### Q3：命令文件放哪里？
**两个位置都可以**：
1、项目级别：`.claude/commands/`（只对当前项目生效）
2、全局级别：`~/.claude/commands/`（对所有项目生效）

老金建议：项目相关的放项目里，通用的放全局。

### Q4：命令太长怎么办？
**拆分成多个小命令，用主命令组合**。

比如：

1、`/deploy-test` 部署到测试环境
2、`/deploy-staging` 部署到预发布
3、`/deploy-prod` 部署到生产

每个都是独立的命令文件，互不干扰。
